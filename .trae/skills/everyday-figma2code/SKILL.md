# Everyday Figma to Code

此 Skill 帮助 AI 编码助手根据 Figma 设计稿，自动生成符合当前项目组件规范、样式方案和架构决策的前端代码。

## 前置条件

1. **Figma MCP Server 已连接**：确保环境中的 Figma MCP Server 处于可用状态，并已提供如 `get_design_context`、`get_screenshot` 等相关工具。
2. **理解与推断项目规范**：如果用户没有明确指明，必须通过阅读项目说明文档或分析已有项目结构，来推断项目的技术栈和框架（通常为 React 或 Vue + TailwindCSS，部分情况可能是原生 HTML）。在开始编写代码前，须先分析并确保代码风格与项目整体保持完全一致，遵循当前项目的组件约定、样式方案和架构模式。

## 必须遵循的实施流程 (Required Flow)

请严格按以下顺序执行操作，**切勿跳过任何步骤**：

1. **获取设计上下文**
   运行 `get_design_context` 工具，获取目标节点（Node）的精确结构化表示。
2. **处理超大响应（如有必要）**
   如果 `get_design_context` 返回的数据过大或被截断，请先运行 `get_metadata` 获取高层级节点映射图，然后再仅对需要的特定节点重新运行 `get_design_context`。
3. **获取视觉参考**
   运行 `get_screenshot`，获取该设计稿节点变体的截图，作为视觉参考。
4. **下载资源与实现准备**
   **只有在同时获得 `get_design_context` 和 `get_screenshot` 后**，才可以下载任何所需的资源并开始代码实现。
5. **翻译为项目规范代码**
   将输出的原始结果（通常为 React + Tailwind 形式的思路）翻译为**符合当前项目约定的框架、组件和样式方案**（例如 Vue、Svelte，或 CSS Modules、styled-components 等）。
   - **组件命名和放置**：遵循当前项目的组件命名规范（如 PascalCase, kebab-case）和放置目录（如 `src/components/`）。
   - **样式与设计 Token**：严禁硬编码颜色值。始终使用项目已定义的设计 Tokens 和间距系统。
6. **视觉与行为验证**
   完成代码编写后，将其与 Figma 截图及交互设计进行对比验证，确保视觉上实现 1:1 的还原并拥有正确的行为交互，确认无误后方可标记为完成。

## 代码编写与组织规范

在生成代码时，请严格遵守以下要求：
- **技术栈与风格**：必须与项目整体技术栈（React/Vue/HTML等）及代码风格保持完全一致。
- **消除重复代码**：还原界面时绝对不要硬写重复代码。对于列表、卡片等动态内容，必须将其抽取为独立的 **mock 数据**、**类型定义 (Types/Interfaces)** 和视图渲染逻辑。
- **组件化拆分**：当页面结构较复杂时，必须将页面合理拆分为多个子组件。
- **布局和已有组件**：优先复用项目中已有的基础 UI 组件。
- **状态与路由**：状态管理如何处理？是否有特定的路由要求？
- **代码导出**：组件导出应采用具名导出还是默认导出？遵循项目的常见模式。
